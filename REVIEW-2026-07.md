## Cross-cutting principles (apply to ALL items)

**P1. Specify a CLASS, not a part (Joshua, 2026-07-10).** The specs define the
requirement envelope and PROVE the path is real (name a class of part that exists +
a few examples so it's demonstrably not vaporware). They do NOT choose the silicon —
that's the hardware firm's job, against availability/cost/their experience.
hardware.md already leans this way ("component choices are recommendations") — hold
that discipline everywhere; where the review names specific parts, treat them as
existence-proof examples of a class, not decisions. Over-commitments that pin a
specific capability (e.g. "USB 3.2 Gen 2", "M.2 NVMe") should become class-level
requirements ("USB3-class host-facing bulk path", "optional high-speed storage bay").

**P2. Separate DESIGN from IMPLEMENTATION SEQUENCING (Joshua, 2026-07-10).** The
design specs (hardware / software-architecture / security) lay out the full
capability envelope — everything the device CAN do. WHICH firmware ships first is an
implementation question that lives in the roadmap/phases, NOT the design docs. So
"is feature X v1 or later?" is almost never a question this document needs to answer:
describe the capability the hardware enables; let phasing handle sequence. Guards
against two failure modes: under-selling (spec describes only what v1 ships) and
over-promising (roadmap items read as shipping features). Keep capability language
("the platform supports…") distinct from commitment language ("v1 ships…").

## A. Show-stoppers — fix before anyone external reads this

### A1. ☐ Backup sync story — DESIGN AGREED 2026-07-10, doc edits pending
- Original problem: "syncs from the drawer" (predecessor's embellishment — not
  Joshua's design). Actual intent: backup syncs **when deliberately powered and
  connected**; if offline, it catches up immediately on next plug-in.
- **Agreed sync paths (make docs tell exactly this story):**
  1. **Direct USB-to-USB** — DRP ports support device-to-device; simplest manual sync.
  2. **Plugged into a computer running the `km` helper** — the USB-Ethernet gadget
     link is point-to-point, so the helper relays/bridges to the LAN (mDNS reflector
     + proxy). Docs must say "computer running the helper," not just "on the network."
     **Network-via-host design (agreed 2026-07-10):** three tiers of host networking:
     - *Bare host, no helper:* point-to-point only. Host can reach the device
       (web UI at the device's link IP); device reaches nothing beyond the host.
       No NTP, no remote sync. State this honestly.
     - *Host with `km` helper (mainline):* helper is a user-space relay needing no
       admin rights — provides (a) time push on enumeration, (b) outbound relay for
       the device's narrow protocol set (NTP, sync to peer KeyMasters), (c) mDNS
       reflection + inbound relay so LAN peers can find and reach the device.
       NOT a general internet gateway — narrow by design.
     - *Power-user path:* OS-level connection sharing (ICS / Internet Sharing /
       Linux NAT) gives the device true routed access. Document, don't default.
     **Anti-PoisonTap spec requirement:** the device's USB-Ethernet gadget must
     NEVER advertise a default route or DNS server in its DHCP offers — link-local
     scope only. A USB device that offers itself as a route is indistinguishable
     from the PoisonTap attack class and will (rightly) get the product flagged by
     enterprise security tooling. Put this in security.md as an explicit rule; it's
     also a selling point for the corporate/IT audience.
  3. **Standalone on the LAN via USB-C Ethernet adapter** — KeyMaster as USB *host*
     on Port B (sourcing power to the adapter) while sinking wall power on Port A.
     Backup becomes a first-class LAN citizen (DHCP + mDNS), no computer needed.
     Two-port design enables this; docs never mention it — add it.
  4. **Safe/vault scenario** — Path 3 + power inside the safe, or PoE splitter
     (one Ethernet cable in → USB-C power + data). Candidate future accessory:
     "backup dock" (PoE → power + network, one plug).
- **New spec requirement — headless/locked sync:** unattended backups must sync
  WITHOUT unlock (nobody enters a PIN in a safe). Cryptographically fine: ciphertext
  replication, device-cert (DRS-bound, not PIN-derived) authentication. But
  hardware.md's power-state table says AP boots on "adequate power + unlock" — needs
  a "designated backup / headless mode" trigger; security.md should state the policy
  and the trade-off (stolen backup keeps receiving ciphertext until unpaired).
- **Headless mode = standalone network appliance (agreed 2026-07-10):** the AP is a
  full Linux box, so a powered + networked backup runs DHCP/mDNS/NTP/kmsyncd with no
  computer involved. Consequences: (a) dead man's switch IS implementable in v1 —
  fix vision.md's journalist scenario from "random ethernet port" to "power +
  Ethernet adapter, or one PoE cable"; (b) NTP gives unattended backups real time.
  Limits to state honestly: independent of a computer, not of power; and network
  time does NOT solve kiosk TOTP on the carried device (A2 still open).
- **Attack-surface rule for headless mode:** while locked, expose ONLY the mutually-
  authenticated sync port — no web UI, no management mDNS. AP is Zone 2; a 24/7
  LAN-resident vault box needs the smallest listening surface. Add to security.md.
- **Doc edits needed:** README:119 ("see each other" → the four paths),
  vision.md:112-114 & 159-164, examples.md:9 (Taylor: backup catches up when plugged
  in, or lives plugged into the family desktop), security.md Tier-1 recovery wording
  ("never more than a sync cycle stale" → "current as of last connection"),
  hardware.md power-state trigger, software.md kmsyncd headless mode.

### A2. ☐ No time source; TOTP is a headline feature
- Batteryless, no RTC, supercap holds ~100ms. In Minimal Mode on an untrusted kiosk
  (no `km` software), the device cannot know the time — so TOTP can't work in the
  flagship scenario. No document mentions timekeeping.
- Dead man's switch time dependency: RESOLVED via A1 headless mode — a powered,
  networked backup gets NTP on its own. Scope of A2 narrows to: **the carried
  primary at an untrusted kiosk** (Minimal Mode, no AP, no network, clock lost on
  unplug — TOTP needs time within ~30s).
- **RESOLVED 2026-07-10 — Joshua approved:**
  1. **Dedicated, relatively large supercap backing an ultra-low-power RTC**
     (e.g., RV-3028, ~45nA timekeeping). Sized generously so the clock survives
     months unpowered (limit is supercap self-discharge, not RTC draw). Keeps
     "batteryless" literally true — marketing line becomes "no battery to charge
     or replace." Add to hardware.md §7 energy-storage table + BOM.
  2. **Opportunistic re-sync from every available source:** NTP whenever the AP is
     up with network; `km` helper / browser extension on any host that has them
     (bare USB provides NO time — host-supplied time always means software).
  2b. **Opportunistic clock scavenging from a bare host (agreed 2026-07-10):**
     over the point-to-point link, probe whatever the environment offers, use the
     best answer, degrade gracefully. Sources, best-to-worst:
     - *NTP query (UDP 123)* — desktops run CLIENTS not servers so a stock
       laptop won't answer, BUT some hosts do (Linux ntpd/chrony in default config,
       corp machines fronting a time server, dev boxes set up as sources). One UDP
       packet to try; when it lands it's the GOLD source — purpose-built, sub-second.
       Always worth attempting.
     - *SMB2 NEGOTIATE response (port 445)* carries the server's full SystemTime —
       full date+time, unauthenticated first handshake packet. Windows hosts when
       the firewall profile permits (often blocked on Public, allowed on
       domain/private). Best trick on corporate Windows.
     - *ICMP Timestamp (type 13)* — ms since midnight UTC: time-of-day but NO DATE.
       Useless if RTC fully dead; good drift-check when RTC alive. Linux/macOS
       usually answer, Windows blocks by default.
     **Probe policy (EDR optics):** an unknown USB NIC peer touching port 445 / 123
     looks like lateral-movement scanning. Probe ONLY when the user requests a TOTP
     code while the clock is unknown/stale, one quiet attempt per source, documented
     openly in the security spec. Best-effort tier only.
     Trust rule unchanged: scavenged time is untrusted → TOTP display only, never
     security timers.
  3. **Manual keypad entry as the anywhere-fallback:** device has 12 keys + display;
     user types current time from watch/phone after long dormancy. TOTP needs only
     ±30s. Clunky but works on any kiosk, fully offline.
  - Note: FAT-timestamp scavenging (host OS writes to an exposed volume carry its
    clock) is cute but unreliable and Minimal Mode exposes no storage — don't spec it.

- **CANONICAL TIME CASCADE (positioning: "opportunistic about time"):**
  1. Supercap-backed RTC — the workhorse; authoritative while charged (months).
  2. NTP proper — when AP is networked (standalone adapter/PoE) or via `km` helper.
     Authoritative.
  3. `km` helper / browser-extension push on enumeration. Authoritative.
  4. Host-clock scavenging (NTP-if-present → SMB → ICMP). Best-effort, untrusted,
     TOTP-display-only.
  5. Manual keypad entry. Universal last resort, works fully offline on any kiosk.
  Frame this as a FEATURE in the pitch: device uses whatever the environment offers
  and degrades gracefully, never hard-depends on one source.

- **Security caveat to spec:** host-supplied time is UNTRUSTED. Fine for TOTP (wrong
  time = codes fail, not compromise) but rate-limit backoff timers must NEVER run on
  host time — an attacker who warps the clock skips the delays. Backoff must use
  internal monotonic/powered time; the hard attempt cap is count-based (already safe).

### A3. ☐ USB/storage architecture — DIRECTION AGREED 2026-07-10, doc edits pending
- **Key insight (Joshua, 2026-07-10):** port FLEXIBILITY/POWER and DATA SPEED are
  separate questions. PD + role negotiation happen on the CC lines, independent of
  USB2-vs-USB3 data. So all the port flexibility below works at USB 2.0 speeds.
- **Can USB be external to the SoC? Depends on role:**
  - *Device/gadget role* (KeyMaster → host): effectively chip-bound. UDC + HS PHY are
    integrated; standalone device-controller chips (MAX3420, FX3) are niche and can't
    cleanly present a fast composite gadget. So the AP's built-in gadget controller
    sets the ceiling for showing storage to the host = USB 2.0 HS on candidate APs.
  - *Hub / host-to-storage-bridge roles*: external chips, already in the design.
    A USB-to-NVMe bridge (e.g. ASMedia ASM2362) CAN give USB3 storage without USB3
    on the AP — but only behind the hub as its OWN mass-storage device, bypassing
    the AP, so it's OUTSIDE the on-device encryption path (non-secret / host-LUKS
    data only). Too complex for v1's small-encrypted-vault crown jewel → Pro/future.
- **DECISIONS:**
  1. KEEP both USB-C ports fully dual-role (data + power) + charge-through.
     NOT gold-plating — three committed use cases each need a different role combo:
     (i) phone + power passthrough (device+source on phone port / sink on wall port);
     (ii) A1 standalone Ethernet-adapter backup (host drives adapter / sink for wall);
     (iii) USB-to-USB direct sync between two units. Justifies the two PD controllers
     on the BOM — good thing to tell a hardware firm.
  2. DESCOPE USB 3.2 + NVMe from v1. Data = USB 2.0 HS (480 Mbps), ample for a
     credential vault; makes i.MX6UL-class candidate APs valid again.
     ** REOPENED 2026-07-10 — see A3-bis below. Inline crypto of external media is a
     real fast-transfer use case that DOES justify USB3 + a modern AP. **
  3. DOCUMENT the USB3-hub + NVMe-bridge path as an explicit Pro/future option WITH
     the caveat that bridged storage sits outside on-device encryption.

### A3-bis. Inline external-media crypto → modern USB3 AP — DECISIONS LOCKED 2026-07-10

  describe the full capability (inline external-media crypto included) as something

- **Per P1, the candidate lists below are EXISTENCE-PROOF EXAMPLES of the class, not

  decrypts inline, presenting plaintext or ciphertext upstream. = open-source
  hardware-encrypted enclosure (Apricorn/iStorage), but multi-purpose. This puts the

- **Open DESIGN work (not sequencing — this genuinely needs specifying):** inline

- **Architecture (holds regardless of AP pick):**
  - KEEP the small dedicated security MCU (M33: STM32U5 / LPC55S69) as its OWN chip.
    Preserves (a) low-power adapter mode (~60mA, AP off — no USB3 SoC runs that low),
    (b) physical isolation of the auditable Zone-0/1 core from the big Linux brain
    (matters for the open-hardware auditability pitch). Collapsing to a single A+M
    die trades physical separation for on-chip TrustZone/enclave isolation — weaker
    argument for this product.
  - Split: M33 = key custody, PIN, vault crypto, always-on. AP = Linux, FUSE, sync,
    and LINE-RATE bulk crypto on its HW engine (M33 too slow for 100s of MB/s).
  - **Key-handoff requirement:** bulk keys must reach the AP crypto engine WITHOUT
    being visible to AP software (Zone 2). NXP CAAM "black keys"/blobs do exactly
    this → a real reason to favor NXP for a security product.
- **Candidate APs (early-2026; USB3 + HW crypto + secure boot), ranked for a
  SECURITY product (crypto/secure-boot maturity + longevity weighted heavily):**
  - *NXP i.MX 8M Plus* — quad A53, 2× USB3 dual-role (host→drive + device→upstream,

    Safe/proven. Cons: multi-watt, big BGA (6-8 layer), unused NPU.
  - *NXP i.MX 95* (2024) — "most current" NXP; A55, USB3, PCIe, EdgeLock secure

### A4. Argon2id 64 MB can't run where crypto runs — RESOLVED + REFRAMED 2026-07-10
- **The contradiction:** security.md sets the PIN-stretch (Argon2id) at 64 MB memory
  "to use the quad-core AP," but crypto runs on the small always-on MCU, which lacks
  that RAM, and in low-power/adapter mode the AP is off entirely → unlock impossible
  as written. (Plain-language: "Argon2id" = a deliberately slow PIN→key recipe;
  making each guess slow is a tax on an attacker guessing offline.)
- **KEY REFRAME (Joshua, 2026-07-10): shift the defense model from "make the math
  slow" (software-PW-manager world) to "let the hardware enforce the limits"
  (hardware-token world — how bank cards / phones / HW wallets actually work). A weak
  PIN is safe not because the number is strong but because the hardware allows few
  guesses and the secret can't be extracted to guess offline.**
- **The defense toolbox (physical/hardware, not computational) — bank as design:**
  1. **Physical-only PIN entry (DECISION):** must physically touch the device; no
     remote/programmatic guessing path. Guessing is hand-speed only.
  2. **Arbitrarily long PIN (DECISION):** supported; length serves coercion/shoulder-
     surf resistance more than brute force (which the hard cap already kills).
  3. **Hard attempt cap enforced by tamper-resistant hardware = the workhorse.** A
     WALL (20 tries ever), not a tax. Counter must be un-resettable without wiping the
     secret. Everything else protects this.
  4. **Un-extractable device secret (DRS) (DECISION) — quality ladder:** flash <
     write-once fuses w/ readout disabled < secure element that does the unlock math
     internally and never emits the secret < (exotic) chip-fingerprint/PUF with
     nothing stored to extract. Forces on-device, capped guessing.
  5. **Erase-on-attack / tamper response (DECISION):** mesh + sensors + supercap-
     backed zeroization so opening/probing DESTROYS the secret first. Converts
     "given a lab + time, extract it" → "the moment they try, it's gone."
  6. **A little stretching as BACKSTOP only:** keep some slowness (light memory-hard
     "if the MCU allows," + iterations) for defense-in-depth if a HW protection fails.
     No longer load-bearing.
- **Honest trade-off to state in security.md:** this converts a MATH problem into a
  PHYSICAL-SECURITY problem — better for ~everyone (physical attacks are costly,
  need the device in hand, don't scale) but it rests on tamper-resistance + secret-
  storage QUALITY. That's why the Pro tier has a real secure element + tamper mesh.
- **DECISIONS FIRMED 2026-07-10 (Joshua): "device destroys itself rather than yield."**
  Use "best" (secure element) or "exotic" (PUF) secret storage; too many wrong
  guesses → supercap-backed WIPE (zeroize the secret, not just lock); full tamper
  resistance that WRECKS the secret store if the case is opened/probed.
  - **The rule that makes self-destruct SAFE, not reckless — state it loudly:**
    aggressive wipe + tamper are acceptable ONLY because data lives in ≥2 places.
    This PROMOTES "sold in pairs" from convenience → SAFETY REQUIREMENT, and makes
    Shamir recovery (A-recovery tiers) load-bearing, not optional. Spec framing:
    "the device destroys itself rather than yield; therefore your secrets live on
    your paired unit and/or recovery shares, never on one device alone." Depends on
    the recovery architecture — do NOT ship aggressive wipe without it.
  - **False-positive design requirement:** tamper meshes/sensors false-trigger (drop,
    cold, supercap brownout, tolerances) — real shipped products have nuked customer
    data this way. Spec MUST (a) name that thresholds be tuned to tolerate real-world
    abuse (firm owns the engineering), (b) rely on recovery as the net for false
    triggers too, not just attacks.
  - **ENCLOSURE DECISION (Joshua, 2026-07-10): STAINLESS STEEL, not aluminum.** Far
    harder to cut/drill/pry; heavier (mass = security here); corrosion-resistant.
    Matches "take real abuse and keep your data." TRADEOFF to design for: stainless
    is a POOR heatsink (~15× worse thermal conductivity than aluminum) → the USB3 AP's
    line-rate-crypto heat gets trapped. Spec must require an INTERNAL THERMAL PATH
    (heat spreader / conductive pad from AP to a dissipation surface). Also harder/
    costlier to machine — fine for a serious product, note in BOM. Replaces
    hardware.md §10 "machined aluminum shell." models/keymaster.scad also says
    aluminum — update or note it's illustrative only.
  - **"wipe" vs old "permanent lockout":** wipe (zeroize DRS/keys) is stronger +
    cleaner than lockout-then-factory-reset. Reconcile the PIN-protection section:
    replace/lockout language with wipe, and point recovery at PAIRED DEVICE / Shamir
    (local ciphertext is intentionally gone after wipe).
  - **Consistency note — SE is the LEAST-open component** (proprietary internals).
    vision.md already handles this honestly (closed SE, everything else auditable) —
    keep that framing; have the "open HW + closed SE" answer ready for reviewers.
  - **SKU implication — tiered security story (update SKU matrix):** base = "good"
    (secret in fuses w/ readout disabled + MCU-level tamper); Pro = "best" (secure
    element + tamper mesh). Both honest, clearly labeled; maps onto the quality
    ladder above. PUF = evaluate-with-firm option (needs error-correction/helper-data
    plumbing — not a checkbox), named as possibility per P1, not a pick.
- **Fix for the Argon2id numbers:** per P1, don't hardcode m/t/p. Specify a
  feasibility target — "PIN-stretch tuned to ~1s on the chosen security-MCU class,
  memory within its on-chip RAM budget" — and demote memory-hardness to backstop.
  Delete the "quad-core AP" reasoning; cross-ref the brute-force section for WHY
  modest stretching suffices (the un-extractable-secret + hard-cap combo does the
  real work). Ties to the section-B "base-model DRS" caveat (name MCU parts with a
  true hardware-secret key, not just a public unique ID).

### A5. Deniability leaks — RESOLVED 2026-07-10 via "object-soup" design (ARBITRARY profiles)
- **The leaks as specced** (any of these lets a flash-image attacker COUNT hidden
  profiles, breaking the "can't prove others exist" promise):
  (a) recipient blobs carry plaintext `profile_id` UUIDs (`security.md:326-331`) —
      count distinct IDs = count profiles.
  (b) per-profile salt records ("device tries all known salts") — count salts.
  (c) plaintext `type: profile|entry|host` tag on stored objects (`software.md:498`)
      — directly labels which objects are profiles.
  (d) total encrypted volume vs. what the revealed profile explains — a soft tell.
  (e) trial-Argon2id across salts → unlock latency scales with profile count (timing).
- **The unavoidable law (state honestly in threat model):** {arbitrary count,
  perfectly hidden count, bounded storage} — pick TWO. Storage is physical → bounded
  is non-negotiable → perfect hiding of an UNbounded count is impossible. Same wall
  TrueCrypt/VeraCrypt hidden volumes hit. Don't overclaim.
- **DECISION (Joshua wants arbitrary, not a fixed slot count): "object-soup" model —
  store NOTHING per-profile that can be counted.** Builds on the content-addressed
  object store the design ALREADY has. Gives EFFECTIVELY ARBITRARY profiles (bounded
  by total storage, not a small fixed N), count genuinely hidden. Mechanics:
  1. ONE device-wide salt, not per-profile (kills leak b; a single salt reveals no
     count). Anti-precomputation still covered by the device-bound secret (DRS).
  2. A profile's root object is LOCATED + keyed by the PIN itself (PIN → derive
     address + key). No PIN → can't find or recognize it. Failed lookup is
     indistinguishable from "no more profiles" (property security.md already wants).
  3. ALL stored blobs uniform-size encrypted, NO plaintext type tag (kills leak c);
     type visible only AFTER decryption (needs PIN).
  4. Sharing = TRIAL-DECRYPT, not labeled recipient lists (kills leak a): entry holds
     a fixed-size bag of anonymous wrapped-keys padded with decoys; device tries each.
  5. Random-fill whole storage at setup so it always looks full (mitigates leak d);
     make duress profile plausibly substantial (already in user-guide guidance).
  6. Constant-work unlock: pad so latency doesn't scale with profile count (leak e).
  - **Joshua confirmed 2026-07-10: spending extra storage on decoys/padding/random-
    fill is FINE — storage is cheap at these scales. Pad liberally.**
- **Two HONEST residual costs (must go in the spec, not hidden):**
  - *Volume tell never hits zero:* attacker can't count profiles but CAN see total
    data volume; sparse duress profile on a stuffed device is a soft (non-proof)
    thread. Random pre-fill + plausible duress shrink it; the "pick two" law means it
    can't be fully eliminated. State it.
  - *Garbage collection is the deep problem:* unlocked as profile A, the device
    CANNOT see profile B's blobs and must NOT delete them (TrueCrypt's trap).
    Requires conservative GC — only remove what the CURRENT profile clearly orphaned,
    never touch unaccounted blobs. Real design surface for security.md + firmware.
- **Concrete edits:** software.md on-disk format — remove plaintext `type` tag,
  replace `recipients:[{profile_id, wrapped_dek}]` with anonymous trial-decrypt bag;
  security.md profile section — one global salt + PIN-derived lookup (delete "tries
  all known salts" list language), add the "pick two" law + residual tells + GC rule.
  Fixed-slot approach (8 slots) is now UNNECESSARY — dropped in favor of object-soup.

### A6. Multi-recipient sharing — RESOLVED 2026-07-10 (private-contacts model)
- **The problem:** sharing wraps the entry key with the destination profile's MASTER
  key — but that only exists inside the recipient's unlocked profile, possibly on
  another device. Can't encrypt to a key you don't have. Cross-profile/cross-device
  sharing (Chen family, spouse Netflix) is impossible as specced.
- **DECISION (Joshua): per-profile PUBLIC/PRIVATE keypair (the X25519 already in the
  primitives table, currently unused), + NO global key directory.** Resolves the A5
  tension: you hold a recipient's public key PRIVATELY inside your own unlocked vault,
  as a "contact." Nothing about who-can-receive is ever published → nothing to
  enumerate → deniability intact. "Must unlock to share" falls out (entry + contacts
  both live behind the PIN).
- **Model:**
  - Each profile: keypair. Public half shareable; private half in the profile root,
    behind the PIN.
  - Public keys exchanged by DELIBERATE PAIRING, never published. In-person: QR on the
    e-paper display (recipient unlocks the profile they want to share INTO, shows its
    pubkey QR, sharer scans → stores as contact). Remote: same exchange over USB/net.
    Hidden profiles never pair → their keys never leave the device = deniability.
  - Share: unlock → pick entry → pick stored contact → seal entry key to their pubkey
    → send sealed blob.
  - Receive: recipient unlocks → their private key opens the sealed blob → entry lands.
- **A5 & A6 ARE THE SAME MECHANISM (spec simplification):** the A5 "anonymous bag of
  wrapped-keys, trial-decrypt, padded with decoys" IS the sharing structure. Each bag
  item = entry key sealed to one recipient's pubkey (own profile OR a contact). No
  labels, trial-decrypt to find yours. Padding hides even HOW MANY share an entry
  (storage is cheap — pad freely). Unify them in the spec; don't build two systems.
- **Transport bonus (reuses A1 network):** sealed blob is opaque ciphertext only the
  recipient opens → transport-agnostic + ASYNC. Ride the sync link OR an untrusted
  relay (email/cloud drop); devices needn't be online simultaneously. Call out as a
  feature.
- **Pairing channels (CORRECTED 2026-07-10 — KeyMaster has NO camera, so it can SHOW
  a QR but not READ one):**
  - *KM ↔ KM: plug one into the other* (USB-to-USB, reuses A1 direct path). Primary
    method, no camera needed. Joshua's instinct — correct.
  - *KM ↔ phone/computer: QR on e-paper*, read by THAT device's camera. QR is only
    for pairing with a camera-equipped device, never KM-to-KM.
  - *Remote: over the network* (rides A1 transport).
  - ALL channels: MITM defense = both devices show a short authentication string
    (words/emoji derived from the exchanged keys); humans confirm they match + press-
    confirm on both keypads (Signal safety-number / BT numeric-comparison pattern).
    Near-zero risk over direct USB; essential for remote.
- **Other design surface (not blockers):** KEY ROTATION/REVOCATION — contact rotates
  key or loses device → past shares need re-sealing. v1 answer: re-pair to update key,
  re-encrypt shared entries on next sync. State it, don't fully solve now.
- **Concrete edits:** software.md crypto — define the per-profile keypair + sealed-box
  (X25519) recipient bag, replacing symmetric-MVK-wrap sharing; security.md — private-
  contacts model + pairing ceremony + rotation note; reconcile crypto-flow.dot and the
  primitives table (X25519 now actually used). user-guide sharing section already
  describes pairing/QR — align wording.

### A7. CAC/PIV wording + passkeys — RESOLVED 2026-07-10
- **CAC/PIV — Joshua's framing accepted, reword for accuracy.** The strong TRUE point:
  the reader/middleware infrastructure does NOT change — it already speaks to PIV
  cards and KeyMaster presents as one; adoption is provisioning + authorization
  (issuer puts certs on the device, trusts it), not re-engineering. Honest spectrum
  to state:
  - *Enterprise / self-run PKI:* can provision KeyMaster TODAY (issue certs, trust
    it). Low bar, true now.
  - *DoD CAC / federal PIV:* technically ready to interoperate with existing readers,
    BUT formal adoption requires the DEVICE to be CERTIFIED (FIPS 140-3, NIST PIV
    test program, GSA APL) + issuer authorization. Slow/expensive business+regulatory
    path — NOT just "key shuffling." This is the "future govt contract" scenario;
    frame as aspirational-but-credible, don't imply it's a near-term checkbox.
  - Reword: "KeyMaster IS a CAC" → "speaks the protocols CAC/PIV infrastructure
    already uses; with issuer authorization + certification it can serve as that
    credential — no reader changes required." (README smart-card section + vision.)
- **Passkeys/FIDO2 — ADD to the capability envelope (Joshua: yes, important).**
  - **Correct the mental model (matters for how to pitch it):** passkeys are NOT
    bearer tokens — they're the OPPOSITE. A passkey is a PRIVATE KEY that never leaves
    the device; auth is challenge-response signing (site sends challenge → device
    signs → only the signature leaves). Nothing replayable crosses the wire, and the
    device refuses to sign for the wrong origin → PHISHING-PROOF. Same family as
    SSH/GPG keys. (Bearer-type secrets KeyMaster also holds: passwords, TOTP seeds,
    wallet seeds — different handling: custody + controlled release vs. never-leaves.)
  - **KeyMaster is an ideal passkey authenticator:** already has HW key storage +
    physical presence (touch) + user verification (PIN). Building blocks all present.
  - **DIFFERENTIATOR — passkey as a vault entry:** store each passkey as an entry type
    (keypair + RP metadata) → it inherits profiles, object-soup deniability, and
    PAIR-AND-SYNC BACKUP. Phones/most YubiKeys can't back up or port passkeys; KM can.
    "Your passkeys, backed up and portable across your devices" = strong, near-unique.
  - **What's actually required (real but well-trodden, not research):** a FIDO2/CTAP2
    authenticator — its own FIDO USB-HID interface (separate from keyboard HID) +
    CTAP2 command set + DISCOVERABLE (resident) credentials (the on-device storage
    that makes a passkey a passkey). Open implementations to learn from: Google
    OpenSK, SoloKeys firmware. Runs on the low-power MCU alongside HID+CCID → passkey
    login WORKS ON AN UNTRUSTED KIOSK in minimal mode (no vault exposed, just a sig).
  - **Edits:** add FIDO2/CTAP2 to software.md interface table + firmware components +
    capability envelope; per P2, firmware phasing is separate. security.md: passkey
    keypairs as vault entries in the key hierarchy. README: keep "passkey store" but
    now it's backed by a real spec.

## B. Second-tier technical corrections

- ☐ **"Secrets never touch the host" (README:149) is false as marketed** — auto-type
  sends plaintext passwords as keystrokes; web UI serves plaintext to the browser.
  True claim: "the vault and keys never touch the host; individual credentials do
  only when you use them."
- ☐ **Budget realism** (`hardware.md:466-477`): firmware MVP at $15–30k is ~3–5× low
  (secure boot + CCID + OpenPGP emulation + EPD UI + touch + vault crypto). Realistic
  dual-processor secure-device prototype: ~$150–400k. Get a real quote or label as
  placeholder.
- ☐ **Nonce math error** (`security.md:243-247`): 96-bit random nonces hit 50%
  collision at ~2^48, not 2^32 (at 2^32 it's ~2^-33). Limit fine; math wrong.
- ☐ **"Version in AAD prevents rollback" — it doesn't.** Replay of whole old
  ciphertext + old AAD works. Needs SE monotonic counter or authenticated index root;
  superblock root_hash is unauthenticated.
- ☐ **Cipher naming drift:** ChaCha20-Poly1305 (security.md) vs XChaCha20-Poly1305
  (hardware.md, crypto-flow.dot). Pick one.
- ☐ **crypto-flow.dot says "no SE in v1"** — contradicts hardware.md (Pro ships SE)
  and security.md (DRS "in SE preferred"). Stale diagram.
- ☐ **Base-model hard-cap caveat:** without SE, attempt counter is in MCU flash.
  Also "DRS from MCU HUK": STM32 unique device ID is PUBLIC; only some parts
  (STM32U5/H5 SAES DHUK) have a real hardware-secret key. Name parts that do.
- ☐ **Low-power current inconsistency:** <20 mA (hardware.md power states) vs <80 mA
  (software.md modes). Also ISO 7816 Class A supplies ~60 mA — state adapter-mode
  budget against it.
- ☐ **Smart-card adapter cannot be passive** (`user-guide.md:407`): USB↔ISO 7816
  needs an active protocol-converter MCU.
- ☐ **Host recognition requires host software** — a USB device can't learn machine
  ID/hostname from enumeration. "Recognizes the client's/hospital's network"
  (examples 7, 11) can't happen on bare kiosks. Spell out the trust chain.
- ☐ **HID auto-type keyboard-layout problem** (scancodes vs non-US layouts; Alt codes
  are Windows-only) — acknowledge. Rephrase "configurable typing speed (evade
  detection)" (`software.md:149`) — bad optics for enterprise/DoD adjacency.
- ☐ **Keypad conflict:** user-guide quick reference reserves 0/*/# for navigation with
  PIN entry on "1-9" → 9-symbol PIN space, contradicting security.md's 10⁶ math and
  arbitrary-length claims.
- ☐ **Permanent-lockout griefing:** a grabber can burn attempts toward lockout.
  Backoff makes it slow (~days) — say so; note Tier-1 recovery is the answer.
- ☐ **KeePassXC "opens XML files directly" — it doesn't** (it opens .kdbx).
  Integration = export/synthesize kdbx or a provider plugin.
